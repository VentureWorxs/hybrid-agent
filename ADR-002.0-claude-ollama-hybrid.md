## 1. Context

ADR-001.1 established the audit log, decision cache, and orchestrator — but offered no way to **measure whether the hybrid architecture is actually working better than baseline**. Token savings are conceptually obvious; proving them requires:

- A way to **turn the hybrid off** (run pure Claude baseline) and **back on** for before/after comparison.
- A way to run audit **without syncing to Cloudflare D1** during sensitive evaluation periods or offline scenarios.
- **KPIs** with clear definitions, calculation rules, and source-of-truth in the audit log.
- A **scorecard** that summarizes those KPIs in formats useful to humans (Markdown), automation (JSON), and Claude Code itself (MCP tool).
- A way to make all of this **automatic over time** — toggling modes should not require manual record-keeping; the audit log should know when modes changed and compare epochs automatically.

This ADR adds the measurement and control plane on top of ADR-001.1's data plane.

## 2. Decision

**Add three layers above the existing audit log:**

### Layer 1 — Operating Modes (three)

- **`baseline`** — All inference routes to Claude API. Ollama never invoked. Used to establish pre-hybrid cost/performance baseline.
- **`hybrid`** — Normal routing per ADR-001.1 (Ollama for routine/sensitive, Claude for complex, agents for domain work). The default.
- **`shadow`** — Same task runs on **both** Ollama and Claude in parallel; primary result returned to user, secondary stored for analysis. Used for rigorous A/B validation. 2x cost during shadow periods.

### Layer 2 — Hierarchical Toggle Resolution

Mode determined by precedence (most-specific wins):

```
Session Override  >  Tenant Override  >  Global Default
```

Plus an **independent** toggle for audit sync:

```
audit_sync_enabled: true | false        (global, with tenant override)
```

When `audit_sync_enabled = false`, events are written to local SQLite with `sync_disabled = 1` and the sync worker skips them. Re-enabling is non-destructive; previously local-only events stay local unless explicitly backfilled.

### Layer 3 — Five Focused KPIs

One KPI per category, deliberately minimal:

| ID | Category | KPI | Direction |
|----|----------|-----|-----------|
| A | Cost / Efficiency | Tokens used (with synthetic all-Claude baseline) | Lower better |
| B | Performance | Throughput (tasks/hour) | Higher better |
| C | Quality / Effectiveness | Approval rate (approvals granted ÷ requested) | Higher better |
| D | Operational | System error rate (errors per 1000 tasks) | Lower better |
| E | Compliance | Boundary enforcement event count | Counted, not minimized |

### Comparison Strategy

- **Epochs** — auto-defined as periods between consecutive `config_changed` events. Each epoch has a unique combination of (operating_mode, audit_sync_enabled, tenant overrides). The scorecard compares any two epochs side-by-side.
- **Synthetic baseline** — for cost comparison, estimate "what would all-Claude have cost?" at the time each Ollama task runs. No actual baseline period required to produce a comparison number.
- **Sample sizes** — reported alongside every KPI. No statistical significance tests in v1; reader judges reliability.

### Scorecard Outputs (Four Formats)

1. **CLI** — `hybrid-agent scorecard ...` prints a formatted table
2. **Markdown** — human-readable report; archivable, shareable
3. **JSON** — structured output for future dashboard or programmatic use
4. **MCP tool** — `get_scorecard` callable from Claude Code mid-session

## 3. Consequences

### Positive

- **Provable claims** — "Hybrid saved X tokens over 30 days" is computable from audit log, not asserted
- **Reversible** — flipping back to baseline is a single config change; no code change, no data loss
- **Offline-capable** — local-only mode lets the agent run anywhere without network dependence on D1
- **Shadow validation** — periodic shadow runs catch quality regressions in Ollama outputs
- **Epoch-aware** — toggle history is part of the audit trail; comparisons stay rigorous even months later
- **Compliance-ready** — boundary enforcement count + sample sizes give Propel-style auditors a defensible number

### Negative / Risks

- **Shadow mode is expensive** — runs every task twice. Should be used in time-bounded campaigns (e.g., one week per quarter), not continuously.
- **Synthetic baseline is an estimate, not ground truth** — token estimation heuristics can drift; documented limitation.
- **Heuristic quality proxies are noisy** — approval rate depends on humans requesting approvals at all. Quiet periods produce small sample sizes.
- **Toggle proliferation** — three layers of override (global / tenant / session) need careful change management or behavior becomes hard to explain.

### Neutral

- KPI list is intentionally small. Adding more later is straightforward (append to KPI catalog, implement calculator). Removing is also fine; metrics not computed simply don't appear in scorecard.
- All scorecard formats derive from the same JSON structure. Adding a dashboard later means reading the same JSON.

## 4. Operating Modes

### 4.1 `baseline`

All inference routes to Claude API. Ollama is never invoked. Other components (decision cache, agents) operate normally — but the cache will only ever store `routed_to: claude-api` decisions.

**Use case**: Establish pre-hybrid baseline cost/performance. Run for 7–14 days to collect a reference epoch, then switch to `hybrid` and compare.

**Routing behavior** in baseline mode:
```python
def route(self, ...):
    return 'claude-api'   # Always
```

The router still produces a `routing_decision` audit event so the epoch's routing pattern is visible.

### 4.2 `hybrid`

Normal routing logic from ADR-001.1:
- Complexity ≤ 3 → `ollama-local`
- Sensitivity `confidential` or `sensitive_phi` → `ollama-local` (boundary enforced)
- Otherwise → `claude-api`
- Domain-specific agents invoked when matched

The default mode after initial baseline collection.

### 4.3 `shadow`

For each subtask:
1. Routing logic produces a **primary** route (same as hybrid mode).
2. The opposite route is invoked as a **shadow** run in parallel.
3. The primary result is returned to the user.
4. The shadow result is stored in the audit log with `is_shadow: true` and a `shadow_pair_id` linking it to its primary.

**Storage pattern**:
```
Event 1: agent_invoked, agent_routed_to='ollama-local', details: {shadow_pair_id: 'pair-001', is_shadow: false}
Event 2: agent_invoked, agent_routed_to='claude-api',  details: {shadow_pair_id: 'pair-001', is_shadow: true}
```

**KPI exclusions**: Shadow events are excluded from primary KPI calculations by default (filter `WHERE NOT json_extract(details, '$.is_shadow') = 1`). Separate shadow-analysis queries pair primary and shadow events for comparison.

**Cost warning**: Shadow mode roughly doubles Claude API cost (since every Ollama task also runs on Claude). The mode controller emits a `config_changed` event with an explicit warning and recommends a campaign window (e.g., `start_at`, `end_at`).

**Recommended shadow campaign**: 7 days, once per quarter, for quality validation.

## 5. Toggle Configuration & Hierarchy

### 5.1 Configuration Sources

| Level | Storage | When applied |
|-------|---------|--------------|
| **Global** | `~/.hybrid-agent/config.toml` | Default for all tenants/sessions |
| **Tenant** | `tenants.metadata` JSON in SQLite | Override per tenant |
| **Session** | In-memory during Claude Code session | Override per session (e.g., explicit user request) |

### 5.2 Config File Schema

`~/.hybrid-agent/config.toml`:

```toml
# Global defaults
[hybrid_agent]
operating_mode = "hybrid"           # baseline | hybrid | shadow
audit_sync_enabled = true            # Push to D1 every 5 min
agent_version = "1.1.0"

[hybrid_agent.shadow_campaign]
# Optional: bound shadow mode to a window
enabled = false
start_at = "2026-06-01T00:00:00Z"
end_at = "2026-06-08T00:00:00Z"

[hybrid_agent.estimation]
# Token estimation rates for synthetic baseline (USD per 1M tokens, Claude rates)
claude_input_cost_per_mtoken = 3.00
claude_output_cost_per_mtoken = 15.00
char_to_token_ratio = 4.0           # Rough chars-per-token estimate
```

### 5.3 Tenant Overrides

Stored in `tenants.metadata` JSON column from ADR-001.1:

```sql
UPDATE tenants
SET metadata = json_set(metadata, '$.operating_mode', 'shadow')
WHERE tenant_id = 'propel';
```

```json
{
  "phi_allowed": true,
  "retention_days": 2555,
  "operating_mode": "shadow",
  "audit_sync_enabled": false
}
```

### 5.4 Session Overrides

Set programmatically at the start of a Claude Code session. Not persisted across sessions. Useful for one-off comparisons or debugging.

```python
# In orchestrator initialization
session_overrides = {
    'operating_mode': 'baseline',   # Force baseline this session
}
```

### 5.5 Resolution Order

```python
def resolve_mode(tenant_id: str, session_overrides: dict | None) -> str:
    if session_overrides and 'operating_mode' in session_overrides:
        return session_overrides['operating_mode']
    
    tenant = get_tenant(tenant_id)
    if tenant.metadata.get('operating_mode'):
        return tenant.metadata['operating_mode']
    
    return global_config.operating_mode
```

Identical logic for `audit_sync_enabled`.

### 5.6 The `config_changed` Audit Event

**Every** toggle change emits a `config_changed` event. This is what defines epochs.

```sql
INSERT INTO audit_events (
    event_id, sequence_number, previous_hash, event_hash,
    tenant_id, machine_id, session_id, agent_version, timestamp,
    event_type, actor, action,
    sensitivity_level, details
) VALUES (
    'ulid...', 142, '...', '...',
    'sam-personal', 'machine-uuid', 'session-uuid', '1.1.0', '2026-05-11T14:23:00Z',
    'config_changed', 'user', 'Operating mode changed: hybrid → baseline',
    'public',
    '{"scope": "global", "field": "operating_mode", "old_value": "hybrid", "new_value": "baseline"}'
);
```

**Required fields in `details`**:
- `scope` — `"global"` | `"tenant"` | `"session"`
- `field` — what changed (`operating_mode`, `audit_sync_enabled`, etc.)
- `old_value`, `new_value` — values before/after
- `tenant_id` — if scope is tenant (redundant with top-level column but explicit)
- `session_id` — if scope is session (same)

## 6. Local-Only Audit Mode

When `audit_sync_enabled = false`:

- Events still write to local SQLite normally
- Events get `sync_disabled = 1` on insert
- The sync worker's query becomes:
  ```sql
  SELECT * FROM audit_events
  WHERE synced_to_d1 = 0 AND sync_disabled = 0
  ORDER BY timestamp ASC
  LIMIT ?
  ```
- Events with `sync_disabled = 1` never leave the machine

### 6.1 Schema Addition

```sql
-- Migration 005_add_sync_disabled.sql
ALTER TABLE audit_events ADD COLUMN sync_disabled INTEGER DEFAULT 0;
CREATE INDEX IF NOT EXISTS idx_events_sync_disabled ON audit_events(sync_disabled, synced_to_d1, timestamp);
```

### 6.2 Re-Enabling Sync

When `audit_sync_enabled` flips back to `true`:

- **Default behavior**: Only new events sync. Existing `sync_disabled = 1` events stay local forever.
- **Backfill option**: `hybrid-agent audit backfill --since=<timestamp>` flips `sync_disabled = 0` on historical events, which the next sync run then pushes to D1.

**Important**: Backfill ignores `sensitivity_level = 'sensitive_phi'` events — those stay confined regardless, per ADR-001.1 sanitization rules.

```python
def backfill_sync(storage, since: datetime, dry_run: bool = True) -> int:
    """
    Mark local-only events as sync-eligible, except sensitive_phi.
    Returns count of events affected. Use dry_run=True first to preview.
    """
    sql = """
        UPDATE audit_events
        SET sync_disabled = 0
        WHERE sync_disabled = 1
          AND sensitivity_level != 'sensitive_phi'
          AND timestamp >= ?
    """
    if dry_run:
        sql = sql.replace("UPDATE audit_events\n        SET sync_disabled = 0",
                          "SELECT COUNT(*) FROM audit_events")
    # ... execute and return count
```

### 6.3 Use Cases for Local-Only Mode

- **Pre-launch testing** — exercise the system without polluting the cloud replica
- **Sensitive evaluation periods** — when running tasks that might contain trade secrets or pre-publication material
- **Offline operations** — long flights, remote locations, intentional network isolation
- **Incident response** — disable sync while investigating a potential compromise

## 7. KPI Catalog

All KPIs derive from `audit_events` in local SQLite. They are computed on demand by the scorecard generator; no separate KPI table is required (though `analytics_*` tables from ADR-001.1 accelerate common queries).

### 7.1 KPI A — Tokens Used (Cost / Efficiency)

**Definition**: Total Claude API tokens consumed in the reporting period, with a parallel synthetic baseline showing what an all-Claude routing would have consumed.

**Source events**: `agent_invoked` events with `tokens_used > 0`.

**Primary calculation**:
```sql
SELECT 
  SUM(CASE WHEN agent_routed_to = 'claude-api' THEN tokens_used ELSE 0 END) AS actual_claude_tokens,
  SUM(CAST(json_extract(details, '$.estimated_claude_tokens') AS INTEGER)) AS estimated_all_claude_tokens
FROM audit_events
WHERE event_type = 'agent_invoked'
  AND tenant_id = ?
  AND timestamp >= ?
  AND timestamp < ?
  AND NOT (json_extract(details, '$.is_shadow') = 1);
```

**Synthetic baseline**: `estimated_all_claude_tokens` is the sum of:
- Actual tokens used (when route = `claude-api`)
- Estimated tokens that would have been used (when route ≠ `claude-api`)

**Synthetic baseline calculation**: See [Section 9](#9-synthetic-baseline-calculation).

**Reported as**:
```
Tokens Used (epoch 7, 30 days, n=1,243 tasks):
  Actual:               245,123 tokens
  Synthetic baseline:   891,456 tokens (if all routed to Claude)
  Savings:              646,333 tokens (72.5%)
  Cost (actual):        $1.84
  Cost (synthetic):     $6.69
  Cost savings:         $4.85
```

**Caveats**:
- Token estimation uses a `char_to_token_ratio` heuristic (default 4.0). Actual ratio varies by content type.
- The "savings" figure is **estimated**, not measured. A baseline-mode epoch provides a measured comparison.

**Sample size threshold**: Report a warning if `n < 50` tasks in period.

### 7.2 KPI B — Throughput (Performance)

**Definition**: Tasks completed per hour during the reporting period.

**Source events**: `task_completed` events.

**Calculation**:
```sql
SELECT 
  COUNT(*) AS task_count,
  MIN(timestamp) AS first_task,
  MAX(timestamp) AS last_task,
  CAST(COUNT(*) AS REAL) / 
    ((julianday(MAX(timestamp)) - julianday(MIN(timestamp))) * 24.0) AS tasks_per_hour
FROM audit_events
WHERE event_type = 'task_completed'
  AND tenant_id = ?
  AND timestamp >= ?
  AND timestamp < ?
  AND NOT (json_extract(details, '$.is_shadow') = 1);
```

**Reported as**:
```
Throughput (epoch 7, 30 days, n=1,243 tasks):
  Average:              1.73 tasks/hour
  Peak hour:            14 tasks/hour
  Active hours:         ~720 of 720 (always-on)
  Avg task latency:     2.1s (median), 4.8s (p95)
```

**Median and p95 latency** (auxiliary stats, included with throughput):
```sql
-- Median (SQLite uses percentile via window functions; D1 may need different approach)
WITH ranked AS (
  SELECT execution_time_ms,
    ROW_NUMBER() OVER (ORDER BY execution_time_ms) AS rn,
    COUNT(*) OVER () AS total
  FROM audit_events
  WHERE event_type = 'task_completed' AND tenant_id = ? AND timestamp >= ? AND timestamp < ?
)
SELECT execution_time_ms AS median FROM ranked WHERE rn = total/2;
```

**Caveats**:
- "Active hours" assumes continuous operation. If the agent was off for part of the period, throughput is underestimated.
- Latency includes Ollama (~300ms) and Claude (~3s) tasks mixed; per-route latency reported as auxiliary stats.

**Sample size threshold**: Report a warning if `n < 50` tasks.

### 7.3 KPI C — Approval Rate (Quality / Effectiveness)

**Definition**: Percentage of human-approval requests that were granted (vs. denied).

**Source events**: `approval_requested`, `approval_granted`, `approval_denied`.

**Calculation**:
```sql
SELECT 
  SUM(CASE WHEN event_type = 'approval_granted' THEN 1 ELSE 0 END) AS granted,
  SUM(CASE WHEN event_type = 'approval_denied' THEN 1 ELSE 0 END) AS denied,
  SUM(CASE WHEN event_type IN ('approval_granted', 'approval_denied') THEN 1 ELSE 0 END) AS total,
  CAST(SUM(CASE WHEN event_type = 'approval_granted' THEN 1 ELSE 0 END) AS REAL) /
    NULLIF(SUM(CASE WHEN event_type IN ('approval_granted', 'approval_denied') THEN 1 ELSE 0 END), 0) * 100 AS approval_rate_pct
FROM audit_events
WHERE tenant_id = ?
  AND timestamp >= ?
  AND timestamp < ?;
```

**Interpretation**:
- **High rate (>90%)** — System escalations are mostly correct; trust score should rise; consider raising auto-approve threshold.
- **Low rate (<60%)** — System is escalating inappropriately, or proposing actions the user doesn't want. Investigate.
- **N too small** — Don't draw conclusions.

**Reported as**:
```
Approval Rate (epoch 7, 30 days):
  Approvals granted:    47
  Approvals denied:     4
  Total requests:       51
  Rate:                 92.2%
  Sample size:          51 ⚠️ small — interpret with caution
```

**Caveats**:
- Heuristic proxy for quality. Low approval rate ≠ Ollama is bad; could mean orchestrator is escalating too aggressively.
- Approval requests are infrequent by design (most decisions auto-approve). Expect small sample sizes.
- A 100% approval rate may simply mean the user always says yes, not that the system is right.

**Sample size threshold**: Report a warning if `n < 20` approval events. Suppress the percentage entirely if `n < 5`.

### 7.4 KPI D — System Error Rate (Operational)

**Definition**: Errors per 1,000 tasks attempted.

**Source events**:
- Numerator: `system_error` + `task_failed` events
- Denominator: `task_started` events

**Calculation**:
```sql
WITH stats AS (
  SELECT 
    SUM(CASE WHEN event_type IN ('system_error', 'task_failed') THEN 1 ELSE 0 END) AS error_count,
    SUM(CASE WHEN event_type = 'task_started' THEN 1 ELSE 0 END) AS task_count
  FROM audit_events
  WHERE tenant_id = ?
    AND timestamp >= ?
    AND timestamp < ?
)
SELECT 
  error_count,
  task_count,
  CAST(error_count AS REAL) / NULLIF(task_count, 0) * 1000 AS errors_per_1000
FROM stats;
```

**Reported as**:
```
System Error Rate (epoch 7, 30 days):
  Errors:               3
  Tasks attempted:      1,246
  Rate:                 2.4 per 1,000 tasks (0.24%)
  Breakdown:
    - ollama timeouts:  2
    - claude rate-limit: 1
```

**Error breakdown** by `agent_routed_to` and error type (from `details.error_type`) provided as auxiliary stat.

**Caveats**:
- Excludes user-caused failures (`task_failed` with `details.cause = 'user_cancelled'`).
- Includes Ollama unavailability — when Ollama service is down, all routed tasks fail. This makes the metric sensitive to local infrastructure issues, which is by design.

**Sample size threshold**: Report a warning if `n < 100` tasks.

### 7.5 KPI E — Boundary Enforcement Event Count (Compliance)

**Definition**: Count of `boundary_enforced` events in the reporting period. **This is a count, not a rate** — the goal is not minimization but visibility.

**Source events**: `boundary_enforced` events.

**Calculation**:
```sql
SELECT 
  COUNT(*) AS enforcement_count,
  SUM(CASE WHEN sensitivity_level = 'sensitive_phi' THEN 1 ELSE 0 END) AS phi_enforced,
  SUM(CASE WHEN sensitivity_level = 'confidential' THEN 1 ELSE 0 END) AS confidential_enforced,
  SUM(CASE WHEN sensitivity_level NOT IN ('sensitive_phi', 'confidential') THEN 1 ELSE 0 END) AS other_enforced
FROM audit_events
WHERE event_type = 'boundary_enforced'
  AND tenant_id = ?
  AND timestamp >= ?
  AND timestamp < ?;
```

**Also report**: Suspected violations (attempts that *should* have been enforced but weren't — anomaly detection):

```sql
-- PHI events that did NOT route to ollama-local — should be 0
SELECT COUNT(*) AS suspected_violations
FROM audit_events
WHERE event_type = 'agent_invoked'
  AND sensitivity_level = 'sensitive_phi'
  AND agent_routed_to != 'ollama-local'
  AND tenant_id = ?
  AND timestamp >= ?
  AND timestamp < ?;
```

**Reported as**:
```
Boundary Enforcement (epoch 7, 30 days):
  Enforcement events:   12
    PHI:                8
    Confidential:       4
    Other:              0
  Suspected violations: 0 ✓
  Sample size:          12 events
```

**Interpretation**:
- A non-zero count with zero violations is **good** — the system is doing its job.
- A zero count is **expected** for tenants without sensitive data (e.g., `sam-personal`).
- **Any** suspected violation is a serious issue — investigate immediately.

**Caveats**:
- For tenants without PHI, this KPI is near-zero by design. The KPI is most meaningful for Propel and tenants with confidential workloads.
- Definition of "suspected violation" depends on `sensitivity_level` being correctly assigned at event creation time.

## 8. Epochs & Comparison Logic

### 8.1 Epoch Definition

An **epoch** is a period during which **all configuration settings are constant**. An epoch begins with a `config_changed` event (or the start of audit log) and ends at the next `config_changed` event (or now).

```sql
-- Find all epoch boundaries for a tenant
SELECT 
  event_id,
  timestamp AS epoch_start,
  json_extract(details, '$.field') AS field_changed,
  json_extract(details, '$.new_value') AS new_value
FROM audit_events
WHERE event_type = 'config_changed'
  AND (tenant_id = ? OR json_extract(details, '$.scope') = 'global')
ORDER BY timestamp ASC;
```

Each epoch is uniquely identified by its starting `config_changed` event_id (or `'genesis'` for the period before any config change).

## 9. Synthetic Baseline Calculation

For KPI A, we need to estimate "what would all-Claude have cost?" for tasks that actually ran on Ollama or other agents.

### 9.1 Per-Event Token Estimation

At Ollama invocation time, compute and store `estimated_claude_tokens` in the event's `details` field.

```python
def estimate_claude_tokens(input_chars: int, output_chars: int, 
                           ratio: float = 4.0) -> dict:
    """
    Estimate Claude token usage from character counts.
    Returns input/output/total token estimates.
    """
    input_tokens = int(input_chars / ratio)
    output_tokens = int(output_chars / ratio)
    return {
        'estimated_input_tokens': input_tokens,
        'estimated_output_tokens': output_tokens,
        'estimated_total_tokens': input_tokens + output_tokens,
    }

def estimate_claude_cost_usd(input_tokens: int, output_tokens: int,
                              input_rate: float = 3.00,  # per million
                              output_rate: float = 15.00) -> float:
    return (input_tokens * input_rate + output_tokens * output_rate) / 1_000_000
```

When `agent_routed_to = 'ollama-local'`, the audit event's `details` includes:
```json
{
  "input_chars": 4200,
  "output_chars": 1850,
  "estimated_claude_tokens": 1512,
  "estimated_claude_cost_usd": 0.0095
}
```

When `agent_routed_to = 'claude-api'`, `estimated_claude_tokens` equals the actual `tokens_used`.

### 9.2 Estimation Limitations

The character-to-token ratio of 4.0 is approximate. Refinements over time:

- **Per-language ratios** — English ~4.0, code ~3.5, Chinese ~1.5. Track in `details.content_type`.
- **Tokenizer truth** — periodically tokenize a sample with `tiktoken` or the actual Claude tokenizer to recalibrate.
- **Calibration via shadow mode** — when in shadow mode, both Ollama and Claude actual token counts are known for the same input. Use the ratio (actual_claude_tokens / chars) to update the global estimate.

### 9.3 Configuration

Estimation parameters live in the config file (Section 5.2). Updating them does not retroactively change historical estimates; new events use the new rates.

When pricing changes (e.g., Anthropic adjusts rates), emit a `config_changed` event with `field = 'estimation.claude_input_cost_per_mtoken'`. This creates an epoch boundary — pre-change and post-change cost figures are compared on like-for-like terms within each epoch.

### 11.1 Mode Controller

```python
# modes/controller.py
import tomllib
from pathlib import Path
from datetime import datetime

class ModeController:
    """Single source of truth for resolved operating mode and sync state."""
    
    def __init__(self, config_path: Path, storage, audit_logger):
        self.config_path = config_path
        self.storage = storage
        self.audit = audit_logger
        self._global_config = self._load_config()
        self._session_overrides = {}
    
    def _load_config(self) -> dict:
        with open(self.config_path, 'rb') as f:
            return tomllib.load(f)
    
    def resolve_mode(self, tenant_id: str) -> str:
        if 'operating_mode' in self._session_overrides:
            return self._session_overrides['operating_mode']
        tenant = self.storage.get_tenant(tenant_id)
        if tenant and tenant.metadata.get('operating_mode'):
            return tenant.metadata['operating_mode']
        return self._global_config['hybrid_agent']['operating_mode']
    
    def resolve_sync_enabled(self, tenant_id: str) -> bool:
        if 'audit_sync_enabled' in self._session_overrides:
            return self._session_overrides['audit_sync_enabled']
        tenant = self.storage.get_tenant(tenant_id)
        if tenant and 'audit_sync_enabled' in tenant.metadata:
            return tenant.metadata['audit_sync_enabled']
        return self._global_config['hybrid_agent']['audit_sync_enabled']
    
    def set_mode(self, mode: str, scope: str, tenant_id: str = None,
                  session_id: str = None, actor: str = 'user') -> None:
        """Change mode and emit config_changed audit event."""
        old_value = self.resolve_mode(tenant_id or 'sam-personal')
        
        if scope == 'global':
            self._global_config['hybrid_agent']['operating_mode'] = mode
            self._persist_global_config()
        elif scope == 'tenant':
            self.storage.update_tenant_metadata(tenant_id, {'operating_mode': mode})
        elif scope == 'session':
            self._session_overrides['operating_mode'] = mode
        
        self.audit.log(
            event_type='config_changed',
            actor=actor,
            action=f"Operating mode: {old_value} → {mode} (scope: {scope})",
            details={
                'scope': scope,
                'field': 'operating_mode',
                'old_value': old_value,
                'new_value': mode,
                'tenant_id': tenant_id,
                'session_id': session_id,
            }
        )
```

### 11.2 Shadow Runner

```python
# modes/shadow_runner.py
import asyncio
from ulid import ULID

class ShadowRunner:
    """Runs primary route and shadow route in parallel; logs both."""
    
    def __init__(self, ollama_agent, claude_agent, audit_logger):
        self.ollama = ollama_agent
        self.claude = claude_agent
        self.audit = audit_logger
    
    async def run_pair(self, task, primary_route: str) -> dict:
        shadow_pair_id = str(ULID())
        shadow_route = 'claude-api' if primary_route == 'ollama-local' else 'ollama-local'
        
        primary_coro = self._run_one(task, primary_route, shadow_pair_id, is_shadow=False)
        shadow_coro = self._run_one(task, shadow_route, shadow_pair_id, is_shadow=True)
        
        primary_result, shadow_result = await asyncio.gather(
            primary_coro, shadow_coro, return_exceptions=True
        )
        
        # User sees the primary; shadow is logged only
        return {'primary': primary_result, 'shadow_pair_id': shadow_pair_id}
    
    async def _run_one(self, task, route: str, pair_id: str, is_shadow: bool):
        agent = self.ollama if route == 'ollama-local' else self.claude
        result = await agent.invoke(task)
        self.audit.log(
            event_type='agent_invoked',
            actor=f"agent:{route}",
            action=f"Executed task (shadow={is_shadow})",
            agent_routed_to=route,
            tokens_used=result.tokens_used,
            cost_usd=result.cost_usd,
            execution_time_ms=result.duration_ms,
            details={
                'shadow_pair_id': pair_id,
                'is_shadow': is_shadow,
                'estimated_claude_tokens': self._estimate_tokens(task, result),
            }
        )
        return result
```

    where_shadow = "" if include_shadow else "AND NOT (json_extract(details, '$.is_shadow') = 1)"
    sql = f"""
        SELECT 
          SUM(CASE WHEN agent_routed_to = 'claude-api' THEN tokens_used ELSE 0 END) AS actual,
          SUM(COALESCE(CAST(json_extract(details, '$.estimated_claude_tokens') AS INTEGER), tokens_used)) AS estimated,
          SUM(CASE WHEN agent_routed_to = 'claude-api' THEN cost_usd ELSE 0 END) AS actual_cost,
          SUM(COALESCE(CAST(json_extract(details, '$.estimated_claude_cost_usd') AS REAL), cost_usd)) AS estimated_cost,
          COUNT(*) AS n
        FROM audit_events
        WHERE event_type = 'agent_invoked' AND tenant_id = ?

    row = storage.execute_fetchone(sql, (tenant_id, start, end))

    calc_tokens_used, calc_throughput, calc_approval_rate,

def generate_scorecard(tenant_id: str, epoch='CURRENT', period=None,
                        format='summary', compare_to=None, 
                        storage=None) -> str | dict:
    primary_epoch = resolve_epoch(storage, tenant_id, epoch, period)
    
    kpis = {
        'tokens_used': calc_tokens_used(storage, tenant_id, primary_epoch.started_at, primary_epoch.ended_at or datetime.utcnow()),
        'throughput': calc_throughput(storage, tenant_id, primary_epoch.started_at, primary_epoch.ended_at or datetime.utcnow()),
        'approval_rate': calc_approval_rate(storage, tenant_id, primary_epoch.started_at, primary_epoch.ended_at or datetime.utcnow()),
        'system_error_rate': calc_system_error_rate(storage, tenant_id, primary_epoch.started_at, primary_epoch.ended_at or datetime.utcnow()),
        'boundary_enforcement': calc_boundary_enforcement(storage, tenant_id, primary_epoch.started_at, primary_epoch.ended_at or datetime.utcnow()),
    }
    
    scorecard_data = {
        'scorecard_version': '1.0',
        'tenant': {...},
        'epoch': primary_epoch.to_dict(),
        'kpis': kpis,
        'comparison': None,
    }
    
    if compare_to:
        compare_epoch = resolve_epoch(storage, tenant_id, compare_to)
        compare_kpis = {...}  # Same calc for comparison epoch
        scorecard_data['comparison'] = compute_deltas(kpis, compare_kpis)
    
    if format == 'json':
        return scorecard_data
    elif format == 'md':
        return render_markdown(scorecard_data)
    else:
        return render_cli(scorecard_data)
